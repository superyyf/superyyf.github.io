devfs#

devfs的出现使得设备驱动程序能自主地管理自己的设备文件，其具有以下优点：

  1. 可以通过程序在设备初始化时在/dev目录下创建设备文件，卸载设备时将它删除。
  2. 设备驱动程序可以指定设备名、所有者和权限位，用户空间程序仍可以修改这些信息。
  3. 不再需要为设备驱动程序分配主设备号以及处理次设备号。

devfs使用范例：

static devfs_handle_t devfs_handle;
static int __init xxx_init(void)
{
    int ret;
    int i;
    /* register device */
    ret = register_chrdev(XXX_MAJOR, DEVICE_NAME, &xxx_fops);
    if(ret < 0){
        ...
    }

    /* create device file */
    devfs_handle = devfs_register(NULL, DEVICE_NAME, DEVFS_FL_DEFAULT, XXX_MAJOR, 0, S_IFCHR | S_IWUSR, &xxx_fops, NULL);
    ...
}

static void __exit xxx_exit(void)
{
    /* delete device file */
    devfs_unregister(devfs_handle);
    /* unregister device*/
    unregister_chrdev(XXX_MAJOR, DEVICE_NAME);
}

module_init(xxx_init);
module_init(xxx_exit);

创建和删除devfs文件节点的API已经被删除了（绝了，最后才告诉我删除了）。

udev用户空间设备管理#

linux 2.6以后，udev取代了devfs，原因如下：

  1. devfs所做的工作被确信可以在用户态来完成。
  2. 发现devfs有一些可修复和无法修复的bug。
  3. devfs的维护者和作者对它感到失望并且已经停止了对代码的维护工作。

Linux设计中强调的一个基本观点是机制和策略的分离。机制是做某样事情的固定步骤、方法。而策略是每一个步骤所采用的不同方法，机制是相对固定的，而每个步骤采用的策略是不固定的。因此，在Linux内核中，不应该实现策略。这就是位于内核空间的devfs应该被用户空间的udev取代的原因，因为devfs管了一些不靠谱的事情。
udev完全在用户态工作，利用设备加入或移除时内核所发送的热插拔事件（Hotplug Event）来工作。在热插拔时，设备的详细信息会由内核通过netlink套接字发送出来（uevent）。udev的设备命名策略、权限控制和事件处理都是在用户态下完成的，它利用从内核收到的信息来进行创建设备文件节点等工作。
Example：

...
fd = socket(PF_NETLINK, SOCK_DGRAM, NETLINK_KOBJECT_UEVENT);
...
while(-1 != poll(...)){
	recv(fd, buf, sizeof(buf), MSG_DONTWAIT);
	...
}

devfs与udev的另一个显著区别是，devfs在/dev节点被打开时自动加载对应的驱动，而udev在设备被发现的时候加载驱动模块。

udev的工作过程#

udev目前和systemd项目合并在一起了，udev的工作过程如下：

  1. 当内核检测到系统中出现了新设备后，内核会通过netlink套接字发送uevent。
  2. udev获取内核发送的消息，进行规则的匹配。匹配的事物包括SUBSYSTEM、ACTION、attribute、内核提供的名称以及其他的环境变量。

udev规则文件#

udev的规则以行为单位，每一行代表一个规则。每一个规则分为一个或多个匹配部分和赋值部分。匹配部分和赋值部分都有专用的关键字来表示。匹配关键字包括：ACTION（行为）、KERNEL（匹配内核设备名）、BUS（匹配总线类型）、SUBSYSTEM（匹配子系统名）、ATTR（属性）等。赋值关键字包括：NAME（创建的设备文件名）、SYMLINK（符号创建链接名）、OWNER（设置设备的所有者）、GROUP（设置设备的组）、IMPORT（调用外部程序）、MODE（节点访问权限）等。
例如：

SUBSYSTEM=="net", ACTION=="add", DRIVERS=="?*", ATTR(address)=="08:00:27:35:be:ff", ATTR(dev_id)=="0x00", ATTR(type)=="1", KERNEL=="eth", NAME="eth1"

这个规则的意思是：当系统中出现的新硬件属于net子系统范畴，系统对该硬件采用的动作是“add”这个硬件，且这个硬件的“address”属性信息等于“08:00:27:35:be:ff”，“dev_id”属性等于“0x00”，“type”属性为1等，此时，对这个硬件采用的动作是创建/dev/eth1。
在嵌入式系统中，也可以用udev的轻量级版本mdev。Android也没有采用udev，它采用vold，机制类似。
